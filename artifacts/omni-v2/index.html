Module 01

Your First Motor Pin

Before we think about four wheels or fancy classes, let's get one motor spinning with the absolute minimum code. The DRV8871 driver takes two PWM inputs (IN1 and IN2). One carrying duty drives the motor one way; the other carrying duty drives it the other way. Both at zero lets the motor coast.

🎯 Learning goals

  • Understand what GPIO pins are and how PWM signals work
  • Understand how two PWM inputs together control direction and speed on a DRV8871
  • Make a single motor spin at ~50% speed with just a few lines of code

What's a pin?

Your Raspberry Pi Pico has physical legs called GPIO pins. We put a PWM signal on them — the pin rapidly switches on and off thousands of times per second. The fraction of time it stays on is the duty cycle. The DRV8871 driver has two PWM inputs, IN1 and IN2:

  • IN1 = duty, IN2 = 0 → motor spins one way
  • IN1 = 0, IN2 = duty → motor spins the other way
  • IN1 = 0, IN2 = 0 → motor coasts to a stop

PWM duty range in MicroPython (duty_u16):

  • duty_u16(0) — always off — no drive
  • duty_u16(32768) — on half the time — ~50% speed
  • duty_u16(65535) — always on — full speed

65535 is the maximum a 16-bit value can hold (that’s 216 − 1). MicroPython uses this range for PWM.

🔌 Wiring diagram — one motor (M1 front-right, DRV8871)
GP11 → IN1 — PWM input (IN1=duty, IN2=0 → spins one way) GP10 → IN2 — PWM input (IN1=0, IN2=duty → spins the other way) VSYS → VCC — 5 V logic power (Pico VSYS carries USB 5 V) GND shared between Pico and driver

ⓘ Connect the motor’s power supply (e.g. a separate battery pack) to the DRV8871 VM pin — do not use VSYS to drive the motor itself.

MicroPython
📄 Replace your whole file with this code
from machine import Pin, PWM
from time import sleep
# M1 front-right: GP11=IN1, GP10=IN2
in1 = PWM(Pin(11))
in2 = PWM(Pin(10))
in1.freq(1000)
in2.freq(1000)
# Spin one way: IN1=duty, IN2=0
in1.duty_u16(32768)
in2.duty_u16(0)
while True:
    sleep(1)

Module 02

Forward, reverse, stop

Typing out the same pin calls every time you want to change direction is repetitive and error-prone. Let's wrap those lines into functions — named chunks of code you can call whenever you need them. No classes yet, just plain functions.

🎯 Learning goals

  • Understand what a function is and why it removes repetition
  • Write motor_clockwise, motor_anticlockwise, and motor_stop
  • Call the functions in sequence with pauses to see each action

How it works

We still use the same two pins from Module 1: IN1 on GP11 and IN2 on GP10, both as PWM outputs. The difference now is that we wrap those calls inside named functions. To go clockwise, set IN2 to a duty value and IN1 to zero. To go anti-clockwise, swap them: IN2 to zero and IN1 to the duty value. To stop, set both to zero.

The duty parameter lets you pass in any value from 0 to 65535, so you can go slow or fast without changing the function itself.

MicroPython
📄 Replace your whole file with this code
from machine import Pin, PWM
from time import sleep
# M1 front-right: GP11=IN1, GP10=IN2
in1 = PWM(Pin(11))
in2 = PWM(Pin(10))
in1.freq(1000)
in2.freq(1000)
def motor_clockwise(duty):
    in1.duty_u16(0)
    in2.duty_u16(duty)
def motor_anticlockwise(duty):
    in1.duty_u16(duty)
    in2.duty_u16(0)
def motor_stop():
    in1.duty_u16(0)
    in2.duty_u16(0)
motor_clockwise(40000)
sleep(2)
motor_anticlockwise(40000)
sleep(2)
motor_stop()

Module 03

Moving in a Direction

One motor works — now let's connect all four. Each DRV8871 driver needs two PWM inputs (IN1 and IN2), so we now have eight PWM objects in total. We'll use the same pattern from Module 2, just scaled up. No classes yet.

🎯 Learning goals

  • Set up eight PWM objects (IN1 + IN2 for each of the four motors)
  • Write functions that move all four wheels together to translate the robot
  • Notice why having eight loose variables starts to feel messy

How it works

For an X-config omni robot, driving forward uses a coordinated four-wheel pattern. On Jay’s tested wiring, M1 and M2 (right side) receive the negative command (a=0, b=duty) while M3 and M4 (left side) receive the positive command (a=duty, b=0). Together, those opposing commands push the robot straight forward instead of spinning. Driving right works the same idea diagonally: opposite pairs of wheels contribute in opposite directions, sliding the robot sideways without turning.

Right now we have eight separate PWM objects (m1a–m4b). It works, but it’s already getting long. Module 4 will show you a much cleaner way.

Motor numbering used in this code: M1 front-right · M2 back-right · M3 back-left · M4 front-left. M1 and M2 are wired on the right side of the robot; M3 and M4 on the left. The forward() function below uses these positions to push the robot straight.
MicroPython
📄 Replace your whole file with this code
from machine import Pin, PWM
from time import sleep
# Motor 1 — front-right  GP11=IN1  GP10=IN2
m1a = PWM(Pin(11)); m1b = PWM(Pin(10))
m1a.freq(1000);     m1b.freq(1000)
# Motor 2 — back-right   GP13=IN1  GP12=IN2
m2a = PWM(Pin(13)); m2b = PWM(Pin(12))
m2a.freq(1000);     m2b.freq(1000)
# Motor 3 — back-left    GP15=IN1  GP14=IN2
m3a = PWM(Pin(15)); m3b = PWM(Pin(14))
m3a.freq(1000);     m3b.freq(1000)
# Motor 4 — front-left   GP18=IN1  GP19=IN2
m4a = PWM(Pin(18)); m4b = PWM(Pin(19))
m4a.freq(1000);     m4b.freq(1000)
def stop_all():
    for p in [m1a, m1b, m2a, m2b, m3a, m3b, m4a, m4b]:
        p.duty_u16(0)
def drive_forward(duty):
    m1a.duty_u16(0);    m1b.duty_u16(duty) # front-right forward contribution
    m2a.duty_u16(0);    m2b.duty_u16(duty) # back-right forward contribution
    m3a.duty_u16(duty); m3b.duty_u16(0)    # back-left forward contribution
    m4a.duty_u16(duty); m4b.duty_u16(0)    # front-left forward contribution
def drive_backward(duty):
    m1a.duty_u16(duty); m1b.duty_u16(0)    # front-right backward contribution
    m2a.duty_u16(duty); m2b.duty_u16(0)    # back-right backward contribution
    m3a.duty_u16(0);    m3b.duty_u16(duty) # back-left backward contribution
    m4a.duty_u16(0);    m4b.duty_u16(duty) # front-left backward contribution
def rotate_clockwise(duty):
    m1a.duty_u16(duty); m1b.duty_u16(0)
    m2a.duty_u16(duty); m2b.duty_u16(0)
    m3a.duty_u16(duty); m3b.duty_u16(0)
    m4a.duty_u16(duty); m4b.duty_u16(0)
def rotate_anticlockwise(duty):
    m1a.duty_u16(0); m1b.duty_u16(duty)
    m2a.duty_u16(0); m2b.duty_u16(duty)
    m3a.duty_u16(0); m3b.duty_u16(duty)
    m4a.duty_u16(0); m4b.duty_u16(duty)
drive_forward(30000)
sleep(2)
drive_backward(30000)
sleep(2)
rotate_clockwise(30000)
sleep(2)
rotate_anticlockwise(30000)
sleep(2)
stop_all()

Module 04

Cleaning Up with Classes

In Module 3 we had eight separate variables just to run four motors. Now we'll replace all of that with a Python class. A class is a blueprint: define it once, then create as many motor objects as you need with a single line each.

🛈 How these lessons work
  • Lessons 1–4 and 6–9 are complete, standalone programs — each one replaces your whole Pico file.
  • Lesson 5 only adds a new class on top of Lesson 4. Keep Lesson 4’s code open and paste Lesson 5’s block below it.
  • Always check the coloured label above each code block — it tells you exactly what to do.

🎯 Learning goals

  • Understand what a Python class is and why it's useful
  • Write a Motor class with __init__ and run methods
  • Replace eight loose variables with four clean Motor objects

How it works

The Motor class stores its own two PWM inputs (IN1 and IN2) inside self. The run(speed) method takes a value from -65535 (full negative) to 65535 (full positive) and handles both inputs automatically. Zero coasts to a stop.

Compare Motor(11, 10) to the eight PWM calls it replaced in Module 3. Positive values drive pin_a and silence pin_b; negative values do the opposite. Which way each wheel actually spins depends on how it is physically wired.

MicroPython
📄 Replace your whole file with this code
from machine import Pin, PWM
import time

class Motor:
    def __init__(self, pin_a, pin_b):
        self.a = PWM(Pin(pin_a))
        self.b = PWM(Pin(pin_b))
        self.a.freq(1000)
        self.b.freq(1000)

    def run(self, speed):
        speed = max(-65535, min(65535, int(speed)))
        if speed > 0:
            # Positive command: IN1=duty, IN2=0
            self.a.duty_u16(speed)
            self.b.duty_u16(0)
        elif speed < 0:
            # Negative command: IN1=0, IN2=duty
            self.a.duty_u16(0)
            self.b.duty_u16(-speed)
        else:
            # Coast to stop: both inputs to 0
            self.a.duty_u16(0)
            self.b.duty_u16(0)

M1 = Motor(11, 10)
M2 = Motor(13, 12)
M3 = Motor(15, 14)
M4 = Motor(18, 19)

all_motors = [M1, M2, M3, M4]
names = ["front-right", "back-right", "back-left", "front-left"]

for motor, name in zip(all_motors, names):
    print("Testing positive command:", name)
    motor.run(30000)
    time.sleep(1)
    print("Testing negative command:", name)
    motor.run(-30000)
    time.sleep(1)
    motor.run(0)
    time.sleep(0.5)

print("All motors tested!")

Module 05

The omni-drive brain: x, y, and rot

This is the heart of the whole robot. Three numbers control everything: x (left/right), y (forward/back), and rot (spin). Four simple equations mix those three values into four individual wheel speeds.

Building on Lesson 4: This code block only contains the OmniRobot class. Keep your Lesson 4 file open and paste it below the M4 = Motor(18, 19) line.

🎯 Learning goals

  • Understand what x, y, and rot mean for an omni robot
  • Learn the four core motor equations
  • Implement drive_xy_rot and test forward, sideways, and spin

The four magic equations

These equations split the robot's desired motion into each wheel's share:

m1 = (−y + x) + rotfront-right
m2 = (−y − x) + rotback-right
m3 = (y − x) + rotback-left
m4 = (y + x) + rotfront-left

When y is positive (forward), the right-side motors (M1, M2) receive a negative command and the left-side motors (M3, M4) receive a positive command. Those opposing commands on each side drive the robot straight instead of spinning.

Map the equations to your robot: each formula line matches one physical motor. m1 = (−y+x)+rot drives the cyan front-right wheel; m3 = (y−x)+rot drives the green back-left wheel. If the robot moves the wrong way, the most likely cause is a motor wired to the wrong position in this diagram.
MicroPython
➕ Keep your Module 4 code open — paste this section below the M4 = Motor(18, 19) line
import time
class OmniRobot:
    def __init__(self, m1, m2, m3, m4):
        self.m1 = m1
        self.m2 = m2
        self.m3 = m3
        self.m4 = m4
    def drive_xy_rot(self, x, y, rot):
        s1 = (-y + x) + rot   # front-right
        s2 = (-y - x) + rot   # back-right
        s3 = ( y - x) + rot   # back-left
        s4 = ( y + x) + rot   # front-left
        self.m1.run(s1)
        self.m2.run(s2)
        self.m3.run(s3)
        self.m4.run(s4)
robot = OmniRobot(M1, M2, M3, M4)
# Forward
robot.drive_xy_rot(0, 30000, 0)
time.sleep(2)
# Backward
robot.drive_xy_rot(0, -30000, 0)
time.sleep(2)
# Right
robot.drive_xy_rot(30000, 0, 0)
time.sleep(2)
# Left
robot.drive_xy_rot(-30000, 0, 0)
time.sleep(2)
# Rotate clockwise
robot.drive_xy_rot(0, 0, 20000)
time.sleep(2)
# Stop
robot.drive_xy_rot(0, 0, 0)

Module 06

Convert angles into movement

The ball sensor gives us an angle — a compass bearing from 0 to 359. We need to turn that into x and y values. That's where sin and cos come in — the same trig functions from math class.

🎯 Learning goals

  • Understand how sin and cos split an angle into x and y components
  • Convert degrees to radians (Python's math library needs radians)
  • Write a drive_angle function that takes an angle and power level

How it works

Think of an angle like the hand of a clock. sin(angle) tells you how far left or right to go, and cos(angle) tells you how far forward or back. Together they give the x and y we pass straight to drive_xy_rot.

x = sin(angle_rad) × power y = cos(angle_rad) × power
MicroPython
📄 Replace your whole file with this code
from machine import Pin, PWM
import math
import time

class Motor:
    def __init__(self, pin_a, pin_b):
        self.a = PWM(Pin(pin_a))
        self.b = PWM(Pin(pin_b))
        self.a.freq(1000)
        self.b.freq(1000)

    def run(self, speed):
        speed = max(-65535, min(65535, int(speed)))
        if speed > 0:
            # Positive command: IN1=duty, IN2=0
            self.a.duty_u16(speed)
            self.b.duty_u16(0)
        elif speed < 0:
            # Negative command: IN1=0, IN2=duty
            self.a.duty_u16(0)
            self.b.duty_u16(-speed)
        else:
            # Coast to stop: both inputs to 0
            self.a.duty_u16(0)
            self.b.duty_u16(0)

class OmniRobot:
    def __init__(self, m1, m2, m3, m4):
        self.m1 = m1
        self.m2 = m2
        self.m3 = m3
        self.m4 = m4

    def drive_xy_rot(self, x, y, rot):
        s1 = (-y + x) + rot   # front-right
        s2 = (-y - x) + rot   # back-right
        s3 = ( y - x) + rot   # back-left
        s4 = ( y + x) + rot   # front-left
        self.m1.run(s1)
        self.m2.run(s2)
        self.m3.run(s3)
        self.m4.run(s4)

def drive_angle(robot, angle_deg, power, rot=0):
    angle_rad = math.radians(angle_deg)
    x = math.sin(angle_rad) * power
    y = math.cos(angle_rad) * power
    robot.drive_xy_rot(x, y, rot)

M2 = Motor(13, 12)
M3 = Motor(15, 14)
M4 = Motor(18, 19)

time.sleep(1)
# 90 degrees = right
drive_angle(robot, 90, 30000)
time.sleep(1)
# 180 degrees = backward
drive_angle(robot, 180, 30000)
time.sleep(1)
# 270 degrees = left
drive_angle(robot, 270, 30000)

Module 07

Read the ball sensor ring

The ball sensor ring is an array of infrared sensors in a circle. It detects the IR-emitting ball and tells us which direction it's at — sending a heading in degrees over UART. We need to read and parse that data.

🎯 Learning goals

  • Understand what a UART serial connection is
  • Read bytes from the sensor ring and decode them into a number
  • Handle the three possible states: no data, ball too close, and heading degrees

How it works

The ball ring sends plain text over UART. On this robot it connects via UART1: TX on GP8, RX on GP9 (Nextion connector). We read a line, strip whitespace, and parse the integer. The sensor returns three kinds of values:

  • None — no data received yet (buffer is empty)
  • -1 — ball is directly under the robot (too close)
  • 0–359 — the angle to the ball in degrees
MicroPython
📄 Replace your whole file with this code
from machine import UART, Pin
import time
ball_uart = UART(1, baudrate=9600, tx=Pin(8), rx=Pin(9))
def read_ball_heading():
    if ball_uart.any():
        try:
            raw = ball_uart.readline()
            if raw is None:
                return None
            decoded = raw.decode("utf-8").strip()
            if decoded == "":
                return None
            return int(decoded)
        except Exception:
            return None
    return None
while True:
    heading = read_ball_heading()
    if heading is None:
        print("No ball data")
    elif heading == -1:
        print("Ball too close")
    else:
        print("Ball at:", heading, "degrees")
    time.sleep(0.1)

Module 08

Add the IMU and hold direction

The IMU (Inertial Measurement Unit) tells the robot which way it's facing. In soccer, other robots will spin yours around. Without heading correction the robot will chase the ball in the wrong direction. The IMU fixes that.

🎯 Learning goals

  • Wire the BNO055 IMU and verify it returns live heading readings
  • Understand why SoftI2C is used instead of hardware I2C on this setup
  • Calculate the shortest angular error between current and target heading
  • Turn that error into a rot correction value for drive_xy_rot

Wiring the BNO055

Connect the BNO055 breakout to the Pico:

  • SDA → GP0
  • SCL → GP1
  • VIN / VCC → 3V3 (the 3.3 V pin on the Pico)
  • GND → GND (shared ground rail)

Why SoftI2C instead of hardware I2C? Hardware I2C can scan the bus and see address 0x28, but then register reads fail with OSError [Errno 5] EIO on this Pico setup. SoftI2C at a slow 10 kHz frequency fixes the problem completely.

Step 1 — Save imu_bno055.py to your Pico

This helper file lives alongside main.py on the Pico filesystem. Copy it once; every lesson that needs the IMU just does from imu_bno055 import create_imu.

MicroPython — imu_bno055.py
💾 Save as imu_bno055.py on the Pico — this is a helper file, not main.py
# imu_bno055.py
# BNO055 IMU driver for Raspberry Pi Pico.
# Uses SoftI2C because hardware I2C can scan 0x28 but then fail
# register reads with OSError [Errno 5] EIO on this setup.
# SoftI2C at 10 kHz is reliable.

from machine import Pin, SoftI2C
import time

# BNO055 register addresses
CHIP_ID_REG  = 0x00
OPR_MODE_REG = 0x3D
PWR_MODE_REG = 0x3E
PAGE_ID_REG  = 0x07
SYS_TRIGGER  = 0x3F
EULER_H_LSB  = 0x1A

MODE_CONFIG  = 0x00   # configuration mode
MODE_NDOF    = 0x0C   # nine-degrees-of-freedom fusion (gives heading)

class BNO055:
    def __init__(self, i2c, address=0x28):
        self.i2c  = i2c
        self.addr = address

    def _write(self, reg, value):
        self.i2c.writeto_mem(self.addr, reg, bytes([value]))

    def _read(self, reg, n=1):
        return self.i2c.readfrom_mem(self.addr, reg, n)

    def begin(self):
        chip_id = self._read(CHIP_ID_REG)[0]
        if chip_id != 0xa0:
            raise RuntimeError("BNO055 chip ID wrong: 0x{:02x}".format(chip_id))
        self._write(OPR_MODE_REG, MODE_CONFIG)
        time.sleep_ms(25)
        self._write(SYS_TRIGGER, 0x20)   # software reset
        time.sleep_ms(650)
        self._write(PWR_MODE_REG, 0x00)  # normal power mode
        self._write(PAGE_ID_REG, 0)      # select page 0
        self._write(OPR_MODE_REG, MODE_NDOF)
        time.sleep_ms(20)

    def heading_deg(self):
        # Euler heading: two bytes LSB-first, unit = 1/16 degree
        data = self._read(EULER_H_LSB, 2)
        raw = data[0] | (data[1] << 8)
        if raw > 32767:
            raw -= 65536
        return raw / 16.0

def create_imu():
    """
    Return a ready-to-use BNO055 on SoftI2C GP0/GP1 at 10 kHz.
    Tries address 0x28 first, then 0x29.
    Uses SoftI2C because hardware I2C may scan 0x28 but fail register
    reads with OSError [Errno 5] EIO on this Pico setup.
    """
    i2c = SoftI2C(sda=Pin(0), scl=Pin(1), freq=10000)
    found = i2c.scan()
    print("I2C scan:", ["0x{:02x}".format(a) for a in found])
    for addr in (0x28, 0x29):
        if addr in found:
            imu = BNO055(i2c, addr)
            imu.begin()
            print("BNO055 ready at 0x{:02x}, chip ID 0xa0".format(addr))
            return imu
    raise RuntimeError("BNO055 not found — check wiring and power")

Step 3 — Heading correction math

The angle_error function finds the shortest rotation needed to reach the target. Without it you might rotate the long way (+270° instead of −90°). The heading_hold_rot function scales that error into a rot value. Positive error → rotate right; negative → rotate left.

MicroPython
📄 Replace your whole main.py — you also need imu_bno055.py on the Pico
import time
from imu_bno055 import create_imu

imu = create_imu()

def angle_error(current, target):
    error = target - current
    while error > 180:
        error -= 360
    while error < -180:
        error += 360
    return error

def heading_hold_rot(current_heading, target_heading, gain=200):
    error = angle_error(current_heading, target_heading)
    rot = error * gain
    rot = max(-30000, min(30000, rot))
    return rot

target_heading = 0   # face 0 degrees (north / forward)

while True:
    current = imu.heading_deg()
    rot = heading_hold_rot(current, target_heading)
    print("Heading: {:.1f}  Rot correction: {:.0f}".format(current, rot))
    time.sleep(0.1)

Module 09

Build the full robot program

This is it — all nine modules combined into one loop. Every 100 ms the robot reads the ball sensor, calculates heading correction, and drives toward the ball while staying oriented forward. This is your playable robot.

🎯 Learning goals

  • Combine Motor, OmniRobot, drive_angle, and UART into one complete program
  • Handle all three ball states cleanly inside the main loop
  • Use IMU heading correction on every single frame

How it works

The loop runs 10 times per second. Each tick:

  1. Read the ball sensor → get the angle to the ball
  2. Read the IMU → get the current facing angle
  3. Calculate rotation correction with heading_hold_rot
  4. If ball found: drive toward it with drive_angle + rot correction
  5. If no ball: hold position and keep correcting heading only

💾 Two files on the Pico: copy imu_bno055.py (from Module 8) and main.py to the Pico before running.

MicroPython
📄 Replace your whole main.py — you also need imu_bno055.py on the Pico
from machine import Pin, PWM, UART
from imu_bno055 import create_imu
import math
import time

class Motor:
    def __init__(self, pin_a, pin_b):
        self.a = PWM(Pin(pin_a))
        self.b = PWM(Pin(pin_b))
        self.a.freq(1000)
        self.b.freq(1000)

    def run(self, speed):
        speed = max(-65535, min(65535, int(speed)))
        if speed > 0:
            # Positive command: IN1=duty, IN2=0
            self.a.duty_u16(speed)
            self.b.duty_u16(0)
        elif speed < 0:
            # Negative command: IN1=0, IN2=duty
            self.a.duty_u16(0)
            self.b.duty_u16(-speed)
        else:
            # Coast to stop
            self.a.duty_u16(0)
            self.b.duty_u16(0)

class OmniRobot:
    def __init__(self, m1, m2, m3, m4):
        self.m1 = m1
        self.m2 = m2
        self.m3 = m3
        self.m4 = m4

    def drive_xy_rot(self, x, y, rot):
        s1 = (-y + x) + rot   # front-right
        s2 = (-y - x) + rot   # back-right
        s3 = ( y - x) + rot   # back-left
        s4 = ( y + x) + rot   # front-left
        self.m1.run(s1)
        self.m2.run(s2)
        self.m3.run(s3)
        self.m4.run(s4)

def drive_angle(robot, angle_deg, power, rot=0):
    angle_rad = math.radians(angle_deg)
    x = math.sin(angle_rad) * power
    y = math.cos(angle_rad) * power
    robot.drive_xy_rot(x, y, rot)

def angle_error(current, target):
    error = target - current
    while error > 180:
        error -= 360
    while error < -180:
        error += 360
    return error

def heading_hold_rot(current_heading, target_heading, gain=200):
    error = angle_error(current_heading, target_heading)
    rot = error * gain
    rot = max(-30000, min(30000, rot))
    return rot

# UART1: ball sensor / Nextion — Pico TX GP8 -> device RX, Pico RX GP9 <- device TX
ball_uart = UART(1, baudrate=9600, tx=Pin(8), rx=Pin(9))

def read_ball():
    if ball_uart.any():
        try:
            raw = ball_uart.readline()
            if raw is None:
                return None
            decoded = raw.decode("utf-8").strip()
            if decoded == "":
                return None
            return int(decoded)
        except Exception:
            return None
    return None

# Motors: M1 front-right, M2 back-right, M3 back-left, M4 front-left
M1 = Motor(11, 10)
M2 = Motor(13, 12)
M3 = Motor(15, 14)
M4 = Motor(18, 19)

robot = OmniRobot(M1, M2, M3, M4)

# IMU: BNO055 via SoftI2C on GP0/GP1 at 10 kHz (imu_bno055.py must be on the Pico)
imu = create_imu()

TARGET_HEADING = 0
DRIVE_POWER = 30000

while True:
    ball = read_ball()
    current_heading = imu.heading_deg()
    rot = heading_hold_rot(current_heading, TARGET_HEADING)
    if ball is None:
        robot.drive_xy_rot(0, 0, rot)
    elif ball == -1:
        robot.drive_xy_rot(0, 0, rot)
    else:
        drive_angle(robot, ball, DRIVE_POWER, rot)
    time.sleep(0.1)

Power & grounding rules
⚠ Motor power must come from a separate battery — not from Pico VSYS.
  • Connect your motor battery positive terminal to VM on each DRV8871.
  • Do not connect the motor battery to Pico VSYS or 3V3 — this will damage the Pico.
  • Connect the motor battery negative (GND) to the shared GND rail alongside Pico GND.
  • All GND pins must share the same rail: Pico GND, each DRV8871 GND, IMU GND, and connector GND.
DRV8871 operation notes
Positive command

Set IN1 to a duty_u16 value and IN2 to zero. The motor runs in one direction.

Negative command

Set IN2 to a duty_u16 value and IN1 to zero. The motor runs in the other direction.

Stop / coast

Set both IN1 and IN2 to zero. The motor coasts freely (not actively braked).

Two-input rule

Only one of IN1 / IN2 carries a duty value at any time. Setting both non-zero simultaneously is invalid and may damage the driver.

Speed control

PWM duty cycle ranges from 0 to 65535 on the Pico. A value of 30 000 is roughly half speed.

Shared ground

Pico GND, every DRV GND, IMU GND, and sensor connector GND must all tie to the same common rail.
